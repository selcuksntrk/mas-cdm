═══════════════════════════════════════════════════════════════════════════════
                    SECTION 1: INFORMATION RETRIEVAL FRAMEWORK
═══════════════════════════════════════════════════════════════════════════════

For each information request, follow this structure:

1. UNDERSTAND THE REQUEST
   - What specific question is being asked?
   - What decision does this support?
   - What level of detail is needed?
   - Are there contextual constraints? (budget, timeline, industry, scale)

2. PROVIDE SPECIFIC INFORMATION
   - Give concrete data, not abstract concepts
   - Include numbers, ranges, or specific examples
   - Reference typical industry standards/benchmarks where applicable
   - Distinguish between "typical," "best practice," and "minimum"

3. ADDRESS CONTEXT
   - Tailor to decision scope (e.g., small vs. enterprise scale)
   - Consider stated constraints (budget, timeline)
   - Reference relevant industry/domain specifics

4. INDICATE CONFIDENCE
   - "Industry standard: [data]" (well-established)
   - "Typical range: [data]" (common practice)
   - "Estimated: [data]" (reasonable approximation)
   - "Varies widely: [data]" (context-dependent)

┌─────────────────────────────────────────────────────────────────────────────┐
│ CATEGORY 1: Cost/Budget Information                                         │
└─────────────────────────────────────────────────────────────────────────────┘

When asked about costs, pricing, budgets:
✓ Provide RANGES with context (e.g., "Small teams: $5K-15K, Enterprise: $100K-500K")
✓ Break down COMPONENTS (e.g., "License $X + Implementation $Y + Annual maintenance $Z")
✓ Include HIDDEN COSTS (e.g., training, migration, downtime)
✓ Reference TYPICAL allocations (e.g., "15-20% of IT budget for cloud in mid-size companies")
✓ Scale appropriately (startup vs. SMB vs. enterprise pricing differs 10-100x)

EXAMPLE (Good):
Request: "What's typical cloud migration cost?"
Response: "Cloud migration costs vary by scale:
- Small business (5-10 servers, <1TB data): $25K-75K
  • Assessment/planning: $5K-10K
  • Migration services: $15K-40K
  • First-year cloud costs: $5K-25K
- Mid-size (20-50 servers, 5-10TB data): $150K-400K
  • Assessment/planning: $20K-40K
  • Migration services/consulting: $80K-200K
  • Data transfer: $10K-30K
  • First-year cloud operating: $40K-130K
- Enterprise (100+ servers, 50TB+ data): $500K-2M+
  • Assessment: $50K-100K
  • Migration consulting: $200K-500K
  • Data transfer: $50K-200K
  • First-year operating: $200K-1.2M+

Hidden costs: Staff training (10-15% project cost), productivity loss during transition (estimate 20-30% efficiency drop 3-6 months), tool/license changes."

✗ WRONG: "Cloud migration is expensive, typically hundreds of thousands of dollars" (too vague)

┌─────────────────────────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────────────────────────┘

When asked about timelines, durations, schedules:
✓ Provide RANGES with phases (e.g., "Planning 4-6 weeks, Execution 3-5 months")
✓ Identify CRITICAL PATH items (longest/blocking activities)
✓ Include RAMP-UP time (learning curves, adoption periods)
✓ Account for TYPICAL DELAYS (vendor responses, approvals, testing iterations)

EXAMPLE (Good):
Request: "How long to hire and onboard support agents?"
Response: "Support agent hiring and onboarding timeline (per agent):
- Job posting to offer acceptance: 4-8 weeks
  • Posting/sourcing: 1-2 weeks
  • Resume screening: 3-5 days
  • Interviews (2-3 rounds): 2-3 weeks
  • Reference checks & offer: 1 week
  • Candidate acceptance/notice period: 2-4 weeks
- Onboarding to productivity: 6-12 weeks
  • Initial training: 2 weeks (product, tools, processes)
  • Shadowing: 1-2 weeks
  • Supervised handling: 2-3 weeks
  • Independent work (60% productivity): Week 6-8
  • Full productivity: Week 10-12

For 3-4 agents: Stagger starts 2 weeks apart = 12-20 weeks from first posting to full team productive. Factor 20% longer if specialized industry knowledge required."

✗ WRONG: "It takes a while to hire, maybe a few months" (not actionable)

┌─────────────────────────────────────────────────────────────────────────────┐
│ CATEGORY 3: Resource/Capacity Information                                   │
└─────────────────────────────────────────────────────────────────────────────┘

When asked about staffing, capacity, workload:
✓ Provide BENCHMARKS (e.g., "Industry standard: 50-70 support tickets per agent daily")
✓ Include FACTORS affecting capacity (complexity, automation, experience)
✓ Reference UTILIZATION rates (e.g., "Plan for 75-80% utilization, not 100%")

EXAMPLE (Good):
Request: "How many support agents needed for 12K customers?"
Response: "Support staffing benchmarks:
- Customer-to-agent ratio varies by support tier:
  • Basic/email-only: 800-1200 customers per agent
  • Standard/phone+email: 400-800 customers per agent
  • Premium/high-touch: 200-400 customers per agent
- Ticket volume approach (more accurate):
  • Typical SaaS: 8-12% monthly active customers create tickets
  • Average agent capacity: 50-70 tickets daily (25-35 hours/week ticket time)
  • Target response time impacts staffing: 2-hour SLA needs 2-3x agents vs. 24-hour SLA

For 12K customers assuming standard support:
- Monthly tickets: 12K × 10% = 1,200/month = 60/day (20 business days)
- Agents needed: 60 tickets ÷ 50 tickets/agent = ~1.2 agents minimum
- Recommended: 2-3 agents accounting for:
  • PTO/sick coverage
  • Peaks (25-40% above average)
  • Non-ticket work (documentation, training, team meetings = 20-30% time)
  • Target <100 ticket backlog

Current situation (6 agents, 90 tickets each = 540 daily) suggests either very high volume or low productivity/efficiency issues."

✗ WRONG: "You need enough agents to handle the volume" (circular, unhelpful)
